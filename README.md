# HomeCleaning-BookingSystem

# 家政保洁预约小程序与后台管理系统 - 需求分析文档

## 一、 C端：用户端（微信小程序）

C端依托微信生态，侧重于提供便捷的上门服务预约、清晰的费用明细、顺畅的支付体验以及透明的退款机制。

### 1. 账户与基础功能
* **授权登录：** 采用微信一键授权登录，静默建立用户档案。
* **个人中心：** 查看历史订单、退款记录。
* **我的优惠券：** 个人中心内设置优惠券卡包，按“未使用”、“已使用”、“已过期”分类展示。
* **地址管理：** 用户可添加、修改、删除服务地址（包含联系人、手机号、省市区、详细门牌号等），下单时需选择对应地址。

### 2. 服务浏览与预约配置 (核心)
* **服务项目列表：** 展示保洁项目（如：日常保洁、深度保洁、家电清洗等），标明计价方式（按小时/按面积/按件）及图文介绍。
* **领券中心：** 在首页或服务列表页增加“领券”入口，展示当前可领取的营销优惠券（如：新人专享券、周末满减券等）。
* **档期与运力选择：**
  * **日历视图：** 展示近期可预约的日期。
  * **时段与运力容量：** 用户选择具体日期后，展示该日期的细分时段（如 09:00-11:00）及其剩余运力（即该时段空闲的保洁员数量）。
* **订单信息录入：** 确认服务地址、选择服务时长/数量，填写特殊备注（如：家里有宠物、需自带某种清洁剂等）。

### 3. 订单与支付
* **创单锁定：** 提交预约后生成待支付订单，并临时锁定该时段运力（建议锁定 10-15 分钟）。
* **优惠券抵扣：** 结算页自动匹配并勾选当前订单可用的最优优惠券，同时支持用户手动更换或取消使用优惠券。页面需实时动态计算并展示“服务总价”、“优惠抵扣金额”以及“实际应付款”。
* **支付网关：** 接入微信支付，直接调用小程序官方支付 API。
* **状态流转：** 微信支付成功回调后，正式扣减运力库存，并同步标记关联优惠券为“已使用”，订单状态变更为“已支付（待派单/待上门）”。超时未支付则订单自动取消，释放运力并退回当前订单已锁定的优惠券。

### 4. 自助取消与退款逻辑 (核心)
* **取消预约入口：** 在订单详情页提供“取消预约并退款”按钮。
* **阶梯退款试算：** 用户点击取消时，系统根据当前时间与预约上门时间的差值，调用后端的阶梯退款规则引擎。实时计算并展示违约金及实际可退金额。
* **自动化退款与退券：** 用户确认后，系统调用微信原路退款接口，订单扭转为“已退款/部分退款”，同时释放该订单占用的运力。
  * **全额退款：** 若产生全额退款，未过期的优惠券将自动退回至用户卡包。
  * **部分退款：** 若因违约扣除部分款项，由于已占用平台资源，使用的优惠券作废不予退还（具体可依后台策略调整）。

## 二、 B端：管理后台系统

B端侧重于保洁项目维护、运力排班配置、动态规则定义和业务履约跟进。

### 1. 运营与营销工作台
* **数据看板：** 核心指标监控，包括今日上门订单数、剩余运力、预计营业额、退款笔数及金额。
* **营销看板：** 监控今日核销优惠券数量、优惠券抵扣总金额等营销数据。

### 2. 资源与运力管理 (核心)
* **保洁项目管理：** 维护服务名称、基础价格、封面图、详情描述、上架状态。
* **排期与运力配置：**
  * 支持按天或按周批量生成排期。
  * 配置指定日期、指定时段的最大可预约运力上限（保洁员数量）。

### 3. 动态退款规则配置 (核心)
* **规则引擎设置：** 提供可视化的阶梯违约金配置界面。定义不同时间节点（距离上门时间的倒计时）对应的违约金扣除比例。
* **规则绑定：** 可将不同的退款规则绑定到不同的保洁项目上。

### 4. 营销与优惠券管理 (核心)
* **卡券模板库：** 创建和管理不同类型的优惠券，如：满减券（满X元减Y元）、折扣券、无门槛代金券。
* **发券规则配置：**
  * **使用条件限制：** 可限制指定保洁项目可用（如仅限日常保洁可用），配置有效期限（支持固定时间段如 10.1-10.7，或动态时间如领取后7天内有效）。
  * **发放方式：** 支持后台定向派发给指定用户手机号、上架到小程序领券中心供用户主动领取，或配置系统行为触发（如新用户注册自动发放新人券）。
* **优惠券明细台账：** 查看所有已发放优惠券的流转状态（未领取、已领取未使用、已使用、已过期），支持追溯核销该券的关联订单号。

### 5. 订单与履约管理
* **预约看板：** 直观展示每天各时段的预约服务明细（含上门地址、客户备注），便于线下调度保洁员。
* **综合订单检索：** 支持按手机号、订单状态、服务日期进行查询。订单详情中需明确记录客户使用的优惠券及实际抵扣的金额。
* **人工干预退款：** 针对客诉，管理员拥有最高权限，可绕过自动规则手动发起全额或部分退款。
* **核销管理 (履约)：** 保洁员上门完成服务后，管理员在后台手动将订单状态核销为“已完成”。

## 三、 技术架构与落地建议

### 1. 前后端技术选型
#### 后端技术栈
* **基础运行环境：** 采用 **C#** 语言，基于最新的 **.NET 8** 平台构建高性能、跨平台的 RESTful API 服务。
* **应用层框架：** 全面引入 **ABP Framework**。利用该框架自带的模块化（Modularity）设计、依赖注入（DI）、基于角色的权限控制（RBAC）、以及开箱即用的工作单元（Unit of Work）和审计日志机制，大幅提升底层基础设施的搭建速度，规范业务代码结构。
* **数据库与 ORM：** 采用 **SQL Server 2016** 作为核心关系型数据库，负责订单、用户档案、优惠券资产及运力排期等关键数据的持久化存储。数据访问层严格采用 **EF Core (Entity Framework Core)** 进行实体映射，并结合 **LINQ to Data** 进行强类型的数据查询与操作，确保防注入安全与开发效率。

#### 前端技术栈
* **C端用户端（微信小程序）：** 明确**目前仅针对微信小程序平台**进行定向开发，不考虑支付宝等其他渠道。技术上建议采用基于 Vue 语法的 **Uni-app** 框架编译为微信小程序（或直接使用微信原生开发），这样既能完美贴合微信生态的授权登录、支付与订阅消息，又能与管理后台的前端技术栈保持高度一致。
* **B端管理后台：** 采用 **Vue.js 3** 配合主流 UI 组件库（如 Element Plus 或 Ant Design Vue）构建单页应用（SPA）。Vue 3 的 Composition API 能够更好地处理后台复杂的表单校验、动态退款规则配置以及数据可视化看板。

### 2. 核心业务场景的落地技术方案

#### (1) 并发控制与防超卖机制 (运力与优惠券)
* 节假日大促（如春节大扫除）易产生瞬时高并发预约。针对“保洁员运力扣减”和“限量优惠券抢领”两大场景，在 **SQL Server 2016** 层面利用 **EF Core 的并发令牌（如 RowVersion 乐观锁）** 来处理并发冲突，确保同一时段的运力或同一张限量优惠券不会被超发。
* 在此基础上，可在架构中预留 Redis 缓存接口，极端高并发下通过 Redis 配合 Lua 脚本进行运力和券库存的“预扣减”，随后再异步落盘到 SQL Server。

#### (2) 业务规则与财务数据的快照设计 (Snapshot)
* 家政服务的价格、退款违约金比例以及优惠券面额可能会随时间动态调整。为保证财务数据的绝对严谨，当用户提交订单并支付时，系统需将当时的“退款规则引擎配置”、“保洁项目原价”以及“优惠券抵扣明细”序列化为 JSON 格式，作为 **硬拷贝（Snapshot）** 直接持久化存储在订单明细表中。
* 未来在处理客诉退款、违约金计算或财务对账时，系统**仅读取该订单关联的快照数据**进行逻辑运算，彻底阻断后台全局规则变更对历史既有订单的干扰。

#### (3) 后台任务调度与自动化流转 (Background Jobs)
* 借助 ABP Framework 内置的**后台工作者服务（Background Workers）**（或集成 Quartz.NET / Hangfire），建立可靠的定时任务调度中心，处理以下自动化业务：
    * **超时未支付处理：** 轮询待支付订单，超时（如15分钟）自动执行“订单取消 -> 释放关联时段的运力 -> 退回已锁定的优惠券”。
    * **优惠券状态巡检：** 每天凌晨定时扫描系统内所有用户的优惠券资产，将超过有效期的优惠券批量更新为“已过期”状态。
    * **服务前提醒：** 距离预约上门时间前 2 小时，自动触发微信小程序订阅消息，提醒用户保洁员即将上门。

#### (4) 履约核销与状态闭环
* 为保证线下服务的真实性，系统不依赖保洁员单方面确认。遵循业务设定，在用户享受完保洁服务后，由后台管理员进行线下回访或确认后，**在管理后台系统中手动将该订单状态核销为“已完成”**。该操作通过 ABP 的工作单元机制，在同一个数据库事务内同步触发保洁员的服务计件统计和平台的营收结算。
